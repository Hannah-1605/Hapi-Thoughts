## CORE BUSINESS RULES — NEVER VIOLATE THESE

These rules are enforced at the system level and must be reflected
in every piece of code you generate:

1. Each Pet belongs to exactly one PetOwner
2. Appointments must go through Admin approval (PENDING → CONFIRMED)
3. Walk-in appointments are created directly as CONFIRMED by Admin
4. Medical records are created and edited by Admin ONLY
5. Medical records are IMMUTABLE — no deletion ever
6. Private notes on medical records are NEVER visible to Pet Owners
7. Billing is generated by Admin — not automatically by the system
8. Payment is handled externally (cash) — Admin records status manually
9. No double-booking — slot exclusivity enforced on every booking
10. Pet Owners can only view public notes on medical records
11. Archived accounts use soft delete — data is NEVER permanently deleted
12. Vaccination records are immutable — use correction notes if wrong
13. Receipt is locked (read-only) once payment status is PAID

## SCOPE RULES — HARD STOP ON THESE

If asked to implement anything from this list, STOP immediately.
Flag it as out of scope and do not proceed without explicit discussion
and approval first:

- Online payment gateway (GCash, Stripe, PayPal)
- SMS notifications (Semaphore, Twilio, any SMS provider)
- Real-time chat or messaging system
- AI-based diagnosis, suggestions, or automation
- Multi-veterinarian scheduling logic
- Multi-clinic or multi-branch support
- Marketplace or e-commerce features
- Receptionist or staff role management
- Inventory management system
- Reporting or analytics module
- Mobile application (iOS/Android)
- WebSockets, Django Channels, or Redis
- Django REST Framework (this is server-rendered, not an API)
- Django built-in /admin interface
- Class-based views

When flagging scope creep, say clearly:
> "This appears to be outside v1 scope as defined in
> BRIEF.md and FEATURES.md. Adding it could introduce scope creep.
> Would you like to discuss whether this is truly needed before
> proceeding?"

## FRONTEND COLLABORATION AWARENESS

The developer has a frontend teammate handling Tailwind CSS styling.
When generating templates, always:
- Write clean semantic HTML first (`<nav>`, `<main>`, `<section>`)
- Add descriptive placeholder class names (e.g. `class="appointment-card"`)
- Add short HTML comments above complex Django template logic
- Keep Django template tags (`{{ }}`, `{% %}`) clearly separated
  from styling concerns
- Never mix business logic into template files

Example of correct template handoff format:
```html
<!-- Appointment card — shows status badge and booking details -->
<div class="appointment-card">
  <span class="status-badge">{{ appointment.get_status_display }}</span>
  <p class="pet-name">{{ appointment.pet.name }}</p>
  <p class="appointment-date">{{ appointment.date }}</p>
</div>
```

The frontend teammate replaces placeholder classes with Tailwind.
She should never need to touch Django template logic.

## WHAT TO WATCH OUT FOR

Proactively flag these common problems before they happen:

**Code cleanliness:**
- If a view is getting too long (>50 lines), suggest splitting it
- If logic is repeated, suggest moving it to utils.py
- If a template is getting complex, suggest breaking it into partials
- Remind the developer to run Black before committing

**Migration safety:**
- Always warn before any model change
- Remind the developer to commit working code before changing models
- Flag if a migration might cause data loss

**Role confusion:**
- Always double-check which role a view is for
- Remind the developer to add role checks at the top of every view
- Flag if an admin view is accidentally accessible to pet owners

**Frontend handoff:**
- Remind the developer to use semantic HTML and placeholder classes
- Flag if Django logic is getting mixed into template styling

**Scope:**
- Flag immediately if a request sounds like it's outside v1 scope
- Suggest adding it to "Known Future Expansion Points" in FEATURES.md
  instead of building it now
